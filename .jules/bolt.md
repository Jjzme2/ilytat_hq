## 2024-05-18 - [Eliminate N+1 transactions on user fetch]
**Learning:** Found a severe bottleneck in `server/api/admin/users.get.ts` where fetching a list of users triggered an N+1 transaction issue. Specifically, any fetched user missing an `employeeId` resulted in a separate transaction updating a single tenant-level metadata counter. When multiple users per tenant were loaded, this led to massive lock contention and retry loops on the counter document.
**Action:** When assigning incrementing IDs to multiple documents (e.g., `employeeId`), avoid N+1 transaction patterns and counter document contention by grouping updates by their parent scope (e.g., `tenantId`) and performing a single transaction per group.
